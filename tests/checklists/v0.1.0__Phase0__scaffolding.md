# Human Test Checklist: Phase 0 — Project scaffolding

**Target version:** v0.1.0
**Ships with commit:** `feat: initial pre-alpha scaffolding for ccipc`
**Companion automated tests:** 8 tests in `tests/test_version.py` (all passing — covers version components, format, PEP 440 compliance, and `ccipc`/`ccipc_lib` re-export consistency).

## What this is

This checklist verifies that the v0.1.0 pre-alpha scaffolding actually *works* on a clean machine — install resolves, both entry points run, the aggregator engine discovers the placeholder tool, the help epilog renders, and the version/release tooling reports the right numbers. **It does NOT verify reverse-compaction functionality** because none is implemented yet (Phase 1 is the first functional release). Anyone running this checklist is verifying the scaffolding's quality — not the project's user-facing value.

Run this before tagging v0.1.0, after merging any PR that touches the package layout, and any time `dazzlecmd-lib` itself bumps a major version.

## Prerequisites

1. **OS:** Windows 11 (primary), Linux (CI/expected), macOS (expected). Tests below note where behavior differs.
2. **Python:** 3.10 or higher.
3. **Tools:** `git`, `pip`. For `dazzlecmd-lib` to resolve, either install it from local source first or have it on PyPI (currently local-only).
4. **Working directory:** A fresh clone of `Claude-Code-Infinite-Perfect-Context` — e.g., `git clone https://github.com/DazzleML/Claude-Code-Infinite-Perfect-Context.git ccipc-test && cd ccipc-test`.
5. **Optional virtualenv:** strongly recommended so install side-effects don't pollute the system Python.

**State isolation note.** Nothing in this checklist writes to `~/.claude/` or any user dotfile. The only writes are to a Python virtualenv site-packages and to the local working tree (e.g., `pip install -e .`).

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. They cover the highest-impact behavior the scaffolding ships. If any fails, the work is broken and the detailed sections below are unlikely to save you.

### HV.1 — Editable install resolves

Validates: `pyproject.toml` declares both packages correctly, `dazzlecmd-lib` dependency resolves, version reporting works end-to-end.

- [ ] **Steps** (cross-shell safe):
  ```
  python -m venv .venv-ccipc-test
  ```
  Then activate (`.venv-ccipc-test\Scripts\activate` on Windows, `. .venv-ccipc-test/bin/activate` on POSIX), and:
  ```
  pip install -e .
  ```
- [ ] **Expected:** Pip succeeds. Final line says `Successfully installed claude-code-infinite-perfect-context-0.1.0` (or `0.1.0.devNN+g<hash>` if version sync added a dev marker). No errors about unresolved `dazzlecmd-lib`.
- [ ] **Why this matters:** If install fails, every other test below is meaningless. This is the smoke test for the entire packaging story.

### HV.2 — Both CLI entry points work

Validates: `[project.scripts]` declarations in `pyproject.toml` produced both shims (`ccipc` and `claude-code-infinite-perfect-context`) and they share state.

- [ ] **Steps** (same across all shells):
  ```
  ccipc --version
  claude-code-infinite-perfect-context --version
  ```
- [ ] **Expected:** Both print the same version string (e.g., `ccipc 0.1.0` or similar). The aliased entry point is just a longer name for the same executable.
- [ ] **Why this matters:** The alias is the project's full name and gives `ccipc` discoverability via PyPI search. If only one works, the alias is broken.

### HV.3 — Help epilog renders the commands and tools sections

Validates: `_build_ccipc_help()` is wired correctly, the `AggregatorEngine` discovers the placeholder tool, and the output mirrors `dz`'s shape.

- [ ] **Steps:**
  ```
  ccipc --help
  ```
- [ ] **Expected:** Output contains a `commands:` section listing `list`, `info <tool>`, `kit`, `version`, `tree`, `setup <tool>`, AND a `core tools:` section containing `search-multi` with a one-line description. The footer reads `Run 'ccipc <tool> --help' for tool-specific options.` and `Pipe stages compose at the shell level: ccipc A | ccipc B | ccipc C`.
- [ ] **Why this matters:** Without the epilog, users can't discover any commands. The epilog is the only thing telling them `ccipc list` and `ccipc info` exist.

### HV.4 — Tool discovery walks the right project root

Validates: `_find_ccipc_project_root()` finds `tools/` and `kits/` correctly whether running from a source checkout OR an installed package.

- [ ] **Steps:**
  ```
  ccipc list
  ccipc info search-multi
  ```
- [ ] **Expected:** `ccipc list` outputs at least one row including `search-multi` under namespace `core`. `ccipc info search-multi` prints the manifest fields (name, namespace, runtime, description, version) without an error.
- [ ] **Why this matters:** The discovery walker is the most fragile piece of the dazzlecmd-lib integration. If it walks from `site-packages` instead of the project root, every tool invocation fails with "tool not found." Test this from both `cd ccipc-checkout && ccipc list` AND `cd /` (or `cd \`) `&& ccipc list` — both should produce the same output.

### HV.5 — All 8 automated tests pass

Validates: regression suite is clean before tagging.

- [ ] **Steps:**
  ```
  python -m pytest tests/ -q
  ```
- [ ] **Expected:** `8 passed` in the summary line. Zero failures, zero errors. Warnings about pytest-asyncio default scope are noise and can be ignored.
- [ ] **Why this matters:** The test suite verifies version-module wiring (PEP 440, dual-package re-export). Failures here indicate the version source-of-truth has drifted.

### HV.6 — Pre-commit version-sync hook fires correctly

Validates: the git hooks installed by `scripts/install-hooks.sh` are wired up so future commits keep `_version.py` and `CHANGELOG.md` in sync.

- [ ] **Steps:**
  ```
  python scripts/sync-versions.py --check
  ```
- [ ] **Expected:** `Version: 0.1.0  (PEP 440: 0.1.0, tag: v0.1.0)` followed by `All versions are in sync.` Exit code 0.
- [ ] **Why this matters:** Out-of-sync version files mean a future tag-and-release will publish a wheel with the wrong version, or pip will resolve it incorrectly. The hook prevents this — if the hook is missing or broken, the pre-commit gate will let bad commits through.

After these six pass, you have high confidence the scaffolding is solid. Detailed sections below cover the rest.

## Section 1: Package layout and imports

### 1.1 ccipc and ccipc_lib are both importable

- [ ] **Steps:**
  ```
  python -c "import ccipc; print(ccipc.__version__)"
  python -c "import ccipc_lib; print(ccipc_lib.__version__)"
  ```
- [ ] **Expected:** Both print the same version string. Neither raises ImportError.

### 1.2 Version re-export is consistent

- [ ] **Steps:**
  ```
  python -c "from ccipc._version import __version__ as a; from ccipc_lib._version import __version__ as b; assert a == b, (a, b); print('ok', a)"
  ```
- [ ] **Expected:** Prints `ok <version>`. AssertionError means the dual-package re-export is broken (this is also what `tests/test_version.py::test_ccipc_re_export` covers — but worth running by hand for the install path).

### 1.3 __main__ entry works

- [ ] **Steps:**
  ```
  python -m ccipc --version
  ```
- [ ] **Expected:** Same version string as `ccipc --version`. This validates `src/ccipc/__main__.py` invokes `cli.main()` correctly.

## Section 2: Aggregator engine integration

### 2.1 search-multi placeholder tool is discoverable but not yet functional

- [ ] **Steps:**
  ```
  ccipc search-multi --help
  ```
- [ ] **Expected:** Argparse output showing the placeholder's flags: `--term`, `--session`, `--limit`, `--explain`, `--type`. The tool is a stub — actually invoking it without args may print a "not yet implemented" message or exit with a non-zero code. **That is correct behavior for v0.1.0.**

### 2.2 Manifest validation

- [ ] **Steps** (Windows PowerShell):
  ```powershell
  Get-Content tools\core\search-multi\.ccipc.json | python -m json.tool
  ```

  **POSIX**:
  ```bash
  cat tools/core/search-multi/.ccipc.json | python -m json.tool
  ```
- [ ] **Expected:** Pretty-printed JSON. Keys include `name`, `namespace`, `runtime`, `script_path`, `description`, `version`. No JSON parse errors.

### 2.3 Kit declaration is well-formed

- [ ] **Steps** (cross-shell):
  ```
  python -m json.tool kits/core.kit.json
  ```
- [ ] **Expected:** Pretty-printed JSON declaring the `core` kit and listing `core:search-multi` (or similar tool reference). No parse errors.

### 2.4 Empty install discovers no extra tools

- [ ] **Steps:**
  ```
  ccipc list
  ```
- [ ] **Expected:** Exactly one tool listed: `search-multi` under `core`. If more tools appear, either someone added them without updating this checklist OR a stale install is leaking from another package.

## Section 5: Cross-shell smoke

### 5.1 Pipeline composition works (using the placeholder)

Even though `search-multi` is a stub, the dispatcher should accept piped invocation without choking on the shell. Tested per-platform:

- [ ] **Steps** (cmd.exe):
  ```cmd
  ccipc list | findstr search-multi
  ```

  **PowerShell**:
  ```powershell
  ccipc list | Select-String search-multi
  ```

  **POSIX**:
  ```bash
  ccipc list | grep search-multi
  ```
- [ ] **Expected:** A line containing `search-multi` is printed. No "broken pipe" errors, no Unicode crashes.

### 5.2 No Unicode/codepage crashes on Windows cmd.exe

Help output should render cleanly on cmd.exe (codepage 437) and PowerShell (codepage 1252).

- [ ] **Steps** (Windows cmd.exe):
  ```cmd
  chcp 437
  ccipc --help
  chcp 1252
  ccipc --help
  ```
- [ ] **Expected:** Both produce readable output. No `?` substitution for ASCII characters, no `UnicodeEncodeError` traceback. (The CLI uses ASCII-only characters in the epilog by design — verify this assumption.)

### 5.3 No Unicode in scripts

- [ ] **Steps** (cross-shell):
  ```
  python -c "import pathlib; [print('NON-ASCII in', f) for f in pathlib.Path('src').rglob('*.py') if any(ord(c) > 127 for c in f.read_text(encoding='utf-8'))]; print('done')"
  ```
- [ ] **Expected:** Just `done`. Any `NON-ASCII in <file>` line indicates a Unicode character snuck into a Python source file (em dashes, smart quotes, arrows). Per CLAUDE.md, Unicode in scripts that may be invoked from cmd.exe is forbidden.

## What the automated tests DON'T cover

These can ONLY be verified by a human running through this checklist:

- **Real `pip install -e .` resolves** — the test suite assumes the package is already installed.
- **Both CLI entry points actually shim correctly** — `pyproject.toml` declarations are not exercised by pytest.
- **`AggregatorEngine` finds the project root from an installed package** — only verifiable on a real install.
- **Help epilog renders correctly across terminals** — terminal width, codepage, color escape handling.
- **Pipeline composition** — `ccipc A | ccipc B` doesn't choke on broken pipes or signal handling.
- **Cross-shell behavior** — cmd.exe vs PowerShell vs bash for the same commands.
- **Pre-commit hook fires correctly** — only happens during a real commit, not in pytest.
- **Manifest JSON validity** — no test parses `tools/core/search-multi/.ccipc.json` independently.
- **GitHub repo configuration** — labels, topics, discussions, and issues #1-#10 only exist on the live repo.
- **ghtraf dashboard renders** — only after GitHub Pages is enabled and the workflow runs.

## Reset / cleanup

After running the checklist:

- [ ] Deactivate the test virtualenv (`deactivate` in either shell).
- [ ] Delete the test virtualenv: `rm -rf .venv-ccipc-test` (POSIX) or `Remove-Item -Recurse -Force .venv-ccipc-test` (PowerShell).
- [ ] If you used a fresh clone, you can delete the test directory entirely.
- [ ] Nothing in `~/.claude/` should have been touched. Verify with `git -C ~ status` if you have the home-repo convention set up.
