# Human Test Checklist: Phase 1 — search-then-fork POC

**Target version:** v0.1.0
**Ships with commit:** TBD (Phase 1 implementation, follows Phase 0 scaffolding `5e03bfc`)
**Companion automated tests:** 108 tests across `tests/unit/` (slug 22, jsonl_search 16, boundaries 8, cost 11, config 11, cc_compat 11) and `tests/integration/` (cli_dispatch 14, fan_in_rejection 4) plus `tests/one-offs/smoke_pipeline.py` end-to-end smoke runner.

## What this is

ccipc Phase 1 ships the search-then-fork POC: 9 library modules (`src/ccipc_lib/`) and 5 CLI tools (`tools/core/<name>/`) that compose into a Unix-pipeline workflow. The tools are dispatched by dazzlecmd-lib's `AggregatorEngine` via `ccipc <tool>` (or `python -m ccipc <tool>`).

This checklist is the human-runnable verification companion to the automated tests. Run it before tagging v0.1.0; run it again after any Phase 2+ refactor that touches dispatch, file I/O, subprocess execution, or metadata handling. Walk through the High-Value section in 5 minutes for a smoke pass; spend ~30 minutes on the detailed sections for a release-gating pass.

## Prerequisites

1. **Python 3.10+** with the project installed: `pip install -e ".[dev]"` from the repo root.
2. **Claude Code installed** AND on PATH (`claude --version` should respond). Required for the slug equality and JSONL-unknown-fields gate tests.
3. **csb (claude-session-backup)** installed: `pip install claude-session-backup`. Used by some tests for backup queries.
4. **A throwaway Claude Code session** to use as a fork target. **Do not test against an active production session** — even though ccipc is read-only against sources, you don't want to clutter your real session list.
5. **Test isolation env vars** — every test in this checklist that runs the tools must use:

   **cmd.exe**:
   ```cmd
   set CLAUDE_HOME=%TEMP%\ccipc_test_claude_home
   set CCIPC_CONFIG_DIR=%TEMP%\ccipc_test_config
   ```

   ```powershell
   $env:CLAUDE_HOME = "$env:TEMP\ccipc_test_claude_home"
   ```

## High-Value Verification (~5 minutes)

If you only have 5 minutes, these six tests cover the highest-impact behaviors. If any fail, the work is broken.

### HV.1 — CLI dispatcher discovers all 5 tools

Validates: dazzlecmd-lib `AggregatorEngine` correctly enumerates tools from `tools/core/*/.ccipc.json` manifests and the `kits/core.kit.json` declaration.

- [ ] **Steps** (same across all shells):

  ```
  ccipc --help
  ```
- [ ] **Expected:** Help output includes `core tools:` section with all 5 names: `search`, `find-boundary`, `cassette`, `cost-estimate`, `hydrate`. Each shows a non-empty description.
- [ ] **Why this matters:** If even one tool is missing, the pipeline can't compose. Manifest discovery and kit filtering must work end-to-end before anything else is meaningful.

### HV.2 — Full 4-stage pipeline runs end-to-end on the synthetic fixture

Validates: search → find-boundary → cassette → cost-estimate composes via the dispatcher, with flat-merge field preservation.

- [ ] **Steps** (POSIX shown; same idea elsewhere):

  ```bash
  ccipc search --session tests/fixtures/synthetic_session.jsonl \
      --term NEEDLE-ALPHA --type user \
    | ccipc find-boundary --before --headroom-tokens 100 --plan api \
    | ccipc cassette --mode A --output /tmp/test_cassette.jsonl --no-inline-meta \
    | ccipc cost-estimate --plan max5 --quiet
  ```
- [ ] **Expected:** Final stdout is one JSONL record containing fields from ALL upstream stages: `matched_terms`, `boundary_uuid`, `cassette_path`, `cost_usd`, `cost_plan_warnings`. Final exit code 0.
- [ ] **Why this matters:** Validates the entire pipeline composes correctly. If this fails, individual stages might still pass but the user-visible workflow is broken.

### HV.3 — Pipeline fan-in rejection (exit 7)

Validates: cassette and hydrate refuse > 1 stdin record with a clear error.

- [ ] **Steps** (POSIX):

  ```bash
  ccipc search --session tests/fixtures/synthetic_session.jsonl \
      --term NEEDLE --type user \
    | ccipc find-boundary --before --headroom-tokens 100 --plan api \
    | ccipc cassette --mode A --output /tmp/should_fail.jsonl --no-inline-meta
  echo "exit=$?"
  ```
- [ ] **Expected:** Exit code 7. Stderr includes "received multiple input records on stdin" and a recovery hint mentioning `head -n 1`. The fixture has 2 user-turn matches for "NEEDLE" so cassette should reject.
- [ ] **Why this matters:** This is the safety gate Gemini surfaced in Round 2. If cassette silently picks the first record, users will get cassettes from the wrong hit with no warning.

### HV.4 — Slug formula matches real CC state (one example)

Validates: ccipc's Python `sanitize_path` reproduces what CC actually wrote to disk.

- [ ] **Steps** (POSIX):

  ```bash
  # Pick any project dir under your real ~/.claude/projects/
  ls ~/.claude/projects | head -1
  # Suppose it shows: C--code-myproject
  # Then derive the cwd from a session inside that project:
  python -c "
  import json, glob, sys
  sys.path.insert(0, 'src')
  from ccipc_lib.slug import slug_from_cwd
  for p in glob.glob('$HOME/.claude/projects/*/[!_]*.jsonl')[:3]:
      with open(p) as f:
          line = f.readline()
      cwd = json.loads(line).get('cwd', '')
      if cwd:
          import os
          dir_actual = os.path.basename(os.path.dirname(p))
          dir_expected = slug_from_cwd(cwd)
          marker = 'OK ' if dir_actual == dir_expected else 'FAIL'
          print(f'{marker} cwd={cwd}')
          print(f'     actual:   {dir_actual}')
          print(f'     expected: {dir_expected}')
  "
  ```
- [ ] **Expected:** Each output line starts with `OK`. If any `FAIL`, the slug formula is drifting from what CC produces.
- [ ] **Why this matters:** If slugs don't match, hydrate writes cassettes to a directory CC won't find on resume. This is the #1 thing CC could break with a future update — needs verification on the user's actual machine before shipping.

### HV.5 — Cost-preview shows plan-aware view FIRST

Validates: the user-facing cost preview prioritizes plan budget over API price.

- [ ] **Steps** (POSIX):

  ```bash
  echo '{"cassette_path": "tests/fixtures/synthetic_session.jsonl"}' \
    | ccipc cost-estimate --plan max5
  ```
- [ ] **Expected:** Stderr shows a "ccipc cost preview" box with **`Plan: max5 -- X% of 5h budget`** as the FIRST line of the body, followed by `API cost: $X.XXXX` second. The plan view is primary; dollar is secondary.
- [ ] **Why this matters:** The user's primary signal is plan budget, not API price. If the order is reversed, Max5/Max20 users miss the right context for their decision.

### HV.6 — Original session JSONL is byte-unchanged after pipeline (soft-touch invariant)

Validates: the pipeline never mutates source data.

- [ ] **Steps** (POSIX):

  ```bash
  # Capture source checksum
  shasum -a 256 tests/fixtures/synthetic_session.jsonl > /tmp/before.sha
  # Run pipeline
  ccipc search --session tests/fixtures/synthetic_session.jsonl \
      --term NEEDLE-ALPHA --type user \
    | ccipc find-boundary --before --headroom-tokens 100 --plan api \
    | ccipc cassette --mode A --output /tmp/cassette.jsonl --no-inline-meta \
    > /dev/null
  # Verify checksum unchanged
  shasum -a 256 tests/fixtures/synthetic_session.jsonl > /tmp/after.sha
  diff /tmp/before.sha /tmp/after.sha && echo "OK: source unchanged"
  ```
- [ ] **Expected:** `OK: source unchanged` printed. The two checksum files diff cleanly (no output from `diff`, just the OK line).
- [ ] **Why this matters:** Soft-touch is a load-bearing invariant. If ccipc ever mutates source state, we've broken the trust contract — users would have to fear running it on real sessions.

After these 6 pass, you have high confidence the work shipped correctly. Detailed sections below cover edge cases, error handling, sidecar correctness, gate behaviors, and the JSONL-unknown-fields gate test that gates a real-CC-execution decision.

## What automated tests DON'T cover (requires this checklist)

The mocks and synthetic fixtures systematically miss:

- **Real `claude --resume` behavior** — the JSONL-unknown-fields gate test (Section 6) is the only way to validate that CC's parser tolerates our `ccipc_meta` injection. Mocks can't simulate CC's parser strictness.
- **Slug equality on the user's actual filesystem** — Section HV.4 hits real `~/.claude/projects/` directories, with whatever historical paths and edge cases the user has accumulated. Synthetic test fixtures use clean paths.
- **TTY vs non-TTY interactive prompts** — pytest captures stdin and stdout differently than a real terminal. Section 3 must be run interactively.
- **Cross-shell command portability** — automated tests run in one process; users invoke ccipc from cmd.exe, PowerShell, bash, zsh, fish. Path-handling, env-var syntax, and quoting all differ.
- **Soft-touch on real source data** — the smoke test verifies checksums on a synthetic fixture, but a paranoid user wants to confirm against THEIR actual session JSONL (HV.6 in extended form).
- **Rendered cost-preview readability** — does the box-drawing show correctly on the user's terminal? Codepage issues, color codes, line wrapping at narrow widths.
- **Performance feel** — does the pipeline feel snappy or sluggish? Are there hangs while CC is being shelled out to?
- **Real CC version compat** — Section 4.3 uses a synthetic version string. The actual SemVer comparison should be re-tested when a new CC version ships.
- **Hidden filesystem behaviors** — junctions on Windows, NFS lag, antivirus interference with `os.replace`, file locks held by other Claude instances.
- **Discoverability** — can a new user, given only `ccipc --help`, figure out how to fork into a past session? The cookbook tries to answer this; running through it as a new user catches gaps the original author can't see.

## Reporting issues found

1. **Note the section and step number** (e.g., "HV.4 failed", "Section 6.1 with cassette X").
2. **Capture the exact command and complete output** (stdout AND stderr).
3. **Save relevant state files** — copy the cassette JSONL, the 3 sidecars, and the source session's first 50 lines.
4. **File an issue** referencing this checklist filename and the version.
5. **Link back** to the triggering commit and CC version (`claude --version`).
