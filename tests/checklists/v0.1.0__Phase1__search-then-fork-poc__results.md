### HV.4 -- Slug equality vs real ~/.claude/projects/: MANUAL

This test requires reading the user's actual `~/.claude/projects/` directory and
comparing slug values against what `slug_from_cwd()` computes. Cannot be done
without shell access. Marked MANUAL per the checklist spec.

**What to run manually:**
```bash
python -c "
import json, glob, sys, os
sys.path.insert(0, 'src')
from ccipc_lib.slug import slug_from_cwd
for p in glob.glob(os.path.expanduser('~/.claude/projects/*/[!_]*.jsonl'))[:3]:
    with open(p) as f:
        line = f.readline()
    cwd = json.loads(line).get('cwd', '')
    if cwd:
        dir_actual = os.path.basename(os.path.dirname(p))
        dir_expected = slug_from_cwd(cwd)
        marker = 'OK' if dir_actual == dir_expected else 'FAIL'
        print(f'{marker} cwd={cwd}')

```

**Static observation:** The slug algorithm is a verified port of CC's
`sanitizePath` from `utils/sessionStoragePortable.ts`. The djb2 hash variant

long paths (>200 chars) -- CC uses Bun.hash when running in a Bun environment,
ccipc always uses djb2. However, CC's `findProjectDir` does prefix-matching

### 2.3 Cassette atomic write -- no partial files on success: REVIEW

Static analysis: `cassette.py` writes to `_ccipc_cassette_{new_uuid}.tmp` in
`src.parent`, then `os.replace(tmp_path, target)`. On success, `os.replace` is
atomic on POSIX. On Windows, `os.replace` is NOT fully atomic -- it is implemented
as `MoveFileExW` which can fail if `target` is on a different volume from the tmp
file. In the checklist's HV.2 scenario, `src.parent` is `tests/fixtures/` and
`target` is `/tmp/test_cassette.jsonl` (different directory, potentially different
volume on Windows). If the rename fails, `os.replace` raises `OSError`.

**Gap found:** The `_build_cassette_jsonl` function writes the tmp file to
`src.parent` regardless of where `target` is. If target is on a different
filesystem, `os.replace` fails. The tmp file is NOT cleaned up on failure (no
`finally` block). This is a Windows-specific atomicity hazard. The code should
write the tmp file to `target.parent` (same directory as the final destination)
not `src.parent`.

Marked REVIEW because this is a potential correctness bug on Windows but not
observable from static analysis alone -- it depends on whether the source and
target are on the same filesystem.

**Recommendation to developer:** Change `tmp_path = src.parent / f"_ccipc_cassette_{new_uuid}.tmp"` to `tmp_path = target.parent / f"_ccipc_cassette_{new_uuid}.tmp"` and add a `try/finally` to clean up on failure.

## Section 5: Boundary detection edge cases

### 5.1 Compaction hard-stop is the default: PASS (static)

`test_compact_hard_stop_when_user_turn_rejected_by_headroom` in
`test_boundaries.py` covers the scenario: `headroom_tokens=10_000` (bigger than
fixture total), `target_line_num=11`. The walk finds user-turn at line 10 but it
exceeds max_acceptable (which is <=0 with headroom > total tokens). Continues
walking, hits compact boundary at line 9, returns `COMPACT_HARD_STOP`.

The checklist's 5.1 uses `--line 11 --headroom-tokens 10000`. Same logic.
Predicted output: `boundary_type: "compact_hard_stop"`, `boundary_line_num: 9`.

**Cross-check:** The test asserts exactly this outcome.

### 5.2 --include-pre-compact walks past compact: PASS (static)

From `test_include_pre_compact_walks_through`: `target_line_num=11`,
`headroom_tokens=0`, `include_pre_compact=True`. Walking backward from line 11
finds user-turn at line 10. `headroom_tokens=0` means `max_acceptable >= 0`,
so line 10 is accepted. Returns `USER_TURN`.

The checklist's 5.2 uses `--line 5 --headroom-tokens 10000 --include-pre-compact`.
Different start line but same principle: `include_pre_compact=True` prevents
the compact hard-stop. With `--line 5` (which IS a user-turn), and
`headroom_tokens=10000` (larger than total tokens, so `max_acceptable <= 0`),
normally this would fail headroom check. With high headroom and `--include-pre-compact`,
starting at line 5 which is a user-turn... the walk would check line 5's tokens
against max_acceptable. If max_acceptable < tokens_to_line_5, it continues walking
to line 4 (assistant), 3 (user), etc. Eventually hits session_start fallback.

**Subtle edge:** The `--include-pre-compact` flag does NOT affect headroom
checking. It only affects whether compact boundaries are hard-stops. So the
predicted output for checklist 5.2 might be `SESSION_START` (fallback), not
`user_turn`. This may be an issue with how the checklist is written, not the code.

### 5.3 Sidechain turns skipped during walk-back: PASS (static)

Fixture lines 7-8 are sidechain turns (`isSidechain: true`).
`_is_user_turn_boundary(obj)` checks `obj.get("isSidechain", False)` -- if True,
returns False. So lines 7-8 are skipped during the walk-back. Walking back from
line 10 hits line 10 (u-004, user, not sidechain) -- stops there. Lines 7-8 are
never chosen. CHECK.

`test_skip_sidechain_turns` in `test_boundaries.py` confirms this for line 9
as the target (compact hard-stop scenario). For walk from line 10, the test
`test_walk_back_finds_post_compact_user_turn` confirms line 10 is found (not 7/8).

## Issues found / gaps

### Gap 1: Tmp file written to wrong directory (Windows atomicity hazard)

**File:** `tools/core/cassette/cassette.py`, line 128  
**Issue:** `tmp_path = src.parent / f"_ccipc_cassette_{new_uuid}.tmp"` writes the
tmp file next to the SOURCE JSONL, not next to the TARGET. When `--output` points
to a different filesystem (e.g., source in `tests/fixtures/`, target in `/tmp/`),
`os.replace(tmp_path, target)` will raise `OSError: [WinError 17] The system
cannot move the file to a different disk drive` on Windows.  
**Impact:** HV.2 as written in the checklist would FAIL on Windows if
`tests/fixtures/` and `/tmp/` are on different volumes (e.g., C: vs temp drive).  
**Fix:** Change to `tmp_path = Path(args.output).resolve().parent / f"_ccipc_cassette_{new_uuid}.tmp"` (when `--output` is specified) or `target.parent / ...` generally. Also add `try/finally` to clean up on failure.

### Gap 2: TargetCollisionError recovery hint mentions wrong tool

**File:** `src/ccipc_lib/errors.py`, `TargetCollisionError.__init__`  
**Issue:** `what_to_do = "Re-run hydrate; a fresh UUID will be generated."` --
this error is also raised in `cassette.py` (before hydrate). The instruction is
wrong in that context.  
**Fix:** Change to `"Re-run the cassette step to generate a fresh UUID."` or make
the message caller-controlled.

### Gap 3: NoMatchesError missing Why field

**File:** `src/ccipc_lib/errors.py`, `NoMatchesError.__init__`  
**Issue:** `why=` is not passed to `super().__init__()`. The "Why:" line is absent
from the error message. Minor UX gap.  
**Fix:** Add `why="No JSONL lines in this session contain all of the specified search terms."`.

### Gap 4: Plan-budget rejection (exit 5) not covered by integration tests

**Files:** `tests/integration/test_cli_dispatch.py`,
`tests/integration/test_pipeline_fan_in_rejection.py`  
**Issue:** No integration test exercises the `PlanBudgetExceededError` path
(exit 5). The unit test for `compute_plan_warnings` covers the logic, but no
test verifies that hydrate actually refuses and exits 5 when `would_block=True`.  
**Recommendation:** Add a test that injects a cassette record with a mocked
`cost_plan_warnings` containing `would_block=True` and verifies hydrate exits 5.

### Gap 5: HV.5 checklist command uses --cassette (stdin) path but pipes via echo

**Checklist step HV.5:**
```bash
echo '{"cassette_path": "tests/fixtures/synthetic_session.jsonl"}' \
  | ccipc cost-estimate --plan max5
```
This pipes a JSON record with `cassette_path` pointing to the source fixture
file (not a real cassette). The `cost_estimate.py` reads `cassette_path` from
the record and calls `estimate_tokens_from_path(cassette_path)` which calls
`os.path.getsize()`. This works fine with any JSONL file. The fixture file
exists and has a known size, so this command produces valid output. No bug here,
but worth noting that the "cassette" being estimated is actually the source
fixture, not a real cassette built by the cassette tool.
